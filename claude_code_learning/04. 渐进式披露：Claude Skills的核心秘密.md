# 渐进式披露：Claude Skills的核心秘密

> 作者：徐公  
> 这是「Claude Code Skills 教程系列」的第 4 篇。本系列将带你从零开始，完整掌握 Claude Code Skills 这一革命性功能。

---

## 一、一个让你意想不到的发现

第1篇文章里，我们提到过"重复输入指令"的痛点。

假设你是一个内容创作者，每次让 Claude 写公众号文章都要输入风格要求：

```Plaintext
写作风格要求：
1. 标题要有吸引力，不超过 20 字
2. 开头用故事或数据引入
3. 正文用小标题分段
4. 语言口语化，避免学术腔
5. 结尾要有互动提问
```

这段指令大约 500 tokens。

如果你每天启动 5 次新对话，一周消耗 17,500 tokens，一年消耗约 900,000 tokens。

**Skills 的解决方案**是：创建一个"公众号写作"技能，第一次配置，之后自动调用。

但你有没有想过一个问题：**如果我有 50 个技能呢？**

每个技能的 SKILL.md 文件平均 3,000 tokens，50 个技能就是 150,000 tokens。

即使 Claude 的上下文窗口很大（比如 200k tokens），这意味着 **75% 的上下文空间**被技能占用，还没开始对话就快满了。

这显然不可持续。

**这时候，Claude Skills 的核心秘密就派上用场了**——它叫"渐进式披露"（Progressive Disclosure）。

理解这个概念后，你会发现：**这是 Skills 能够大规模应用的关键设计**。

---

## 二、什么是渐进式披露？

### 从用户体验说起

"渐进式披露"其实不是 AI 领域的原创概念，它来自用户体验设计。

想象一下，如果 Photoshop 把所有功能都放在界面上：

- 100+ 个菜单项
- 500+ 个工具按钮
- 3000+ 个参数设置

结果会是什么？**新手直接吓跑，老用户也找不到需要的功能**。

所以优秀软件的做法是：

- **基础功能**：默认可见
- **高级功能**：按需展开
- **专业选项**：藏在设置里

这就是渐进式披露——**需要时才给信息，别一次性全扔出来**。

### Claude Skills 的应用

Anthropic 把这个概念用到了 Skills 的设计上：

**传统方式的问题**：

- 把所有技能的完整内容都加载进上下文
- 即使你只需要用其中 1-2 个技能
- 大量 tokens 被浪费

**渐进式披露的解决方案**：

- 分三级加载技能内容
- 需要时才触发
- 初始 token 消耗大幅降低

这个设计看起来简单，但很精妙。

---

## 三、三级加载机制详解

这是本文的核心部分，我们深入剖析这个机制的技术原理。

### 第一级：元数据（始终加载）

**触发时机**：Claude 启动时

**加载内容**：只加载每个技能的元数据部分，也就是 SKILL.md 文件顶部的 YAML 格式信息：

```YAML
---
name: prd-generator
description: 将需求整理成规范的产品需求文档，适用于产品经理和项目负责人
---

# PRD 生成器

（以下内容暂不加载）
```

**Token 消耗**：每个技能约 **100 tokens**

**工作原理**：

- Claude 扫描 `.claude/skills/` 目录
- 读取每个技能的 `name` 和 `description`
- 把这些信息存入"技能索引"

**实际效果**：

- 即使你有 50 个技能，初始加载只消耗 **5,000 tokens**
- 在 200k tokens 的上下文窗口中，占比 **2.5%**

**这一级的作用**：让 Claude "知道"有哪些技能可用，但不知道怎么用。

---

### 第二级：SKILL.md 主体（触发时加载）

**触发时机**：Claude 判定某个技能与当前任务相关时

**加载内容**：完整的 SKILL.md 文件（包括 YAML 元数据 + Markdown 主体内容）

**Token 消耗**：平均每个技能 **2,000-5,000 tokens**

**触发机制示例**：

```Plaintext
用户输入：帮我把会议记录整理成 PRD

Claude 的判断过程：
1. 扫描第一级元数据（已在内存中）
2. 匹配到 "prd-generator" 的描述：
   "将需求整理成规范的产品需求文档..."
3. 判定：这个技能与当前任务高度相关
4. 触发第二级加载：读取完整的 SKILL.md
5. 执行：按照文件中的指令生成 PRD
```

**官方推荐**：SKILL.md 主体内容控制在 **500 行以内**，约 **5k tokens 以下**

**这一级的作用**：让 Claude "学会"如何使用这个技能。

---

### 第三级：打包资源（按需加载）

**触发时机**：Claude 执行任务时，确实需要某个参考文件

**加载内容**：references/ 目录下的具体文件

**Token 消耗**：按实际文件大小

**触发机制示例**：

假设你的 `prd-generator` 技能结构如下：

```Plaintext
prd-generator/
├── SKILL.md
└── references/
    ├── prd-template.md          # PRD 模板
    ├── requirement-checklist.md # 需求检查清单
    └── examples/
        ├── saas-prd-sample.md   # SaaS 产品案例
        └── feature-prd-sample.md # 功能型案例
```

在 SKILL.md 中，你可以这样引用：

```Markdown
## PRD 模板
详见 [PRD_TEMPLATE.md](references/prd-template.md)

## 案例参考
- SaaS 产品案例：[EXAMPLE_SAAS.md](references/examples/saas-prd-sample.md)
- 功能型案例：[EXAMPLE_FEATURE.md](references/examples/feature-prd-sample.md)
```

**Claude 的执行流程**：

```Plaintext
用户：帮我把会议记录整理成 PRD

Claude：
1. 加载第二级（SKILL.md 主体）
2. 执行到"使用 PRD 模板"这一步
3. 发现需要读取 prd-template.md
4. 触发第三级加载：读取该文件
5. 基于模板生成 PRD

（如果某个案例未被使用，则不会加载对应的案例文件）
```

**这一级的作用**：提供"无限容量"的扩展空间，不影响初始加载。

---

### 三级机制的技术优势

让我们用表格对比：

| 级别 | 加载时机 | Token 消耗 | 技能数量影响 |
|-|-|-|-|
| **第一级** | 启动时 | \~100 tokens/技能 | 线性增长（50技能=5k tokens） |
| **第二级** | 按需触发 | \~3,000 tokens/技能 | 只加载相关的 1-2 个 |
| **第三级** | 具体使用 | 按文件大小 | 只用到的才加载 |

**关键洞察**：

- 第一级：**低固定成本**，可以放心配置大量技能
- 第二级：**按需边际成本**，只在实际使用时才产生消耗
- 第三级：**无限制扩展**，复杂度不影响初始性能

## 四、Token 效率对比：数据说话

让我们用具体数字对比传统方式和渐进式披露。

### 场景设定

假设你有这 5 个技能：

| 技能名称 | SKILL.md 大小 |
|-|-|
| TypeScript 代码规范 | 2,500 tokens |
| PRD 生成器 | 1,800 tokens |
| 公众号写作助手 | 2,200 tokens |
| 数据库查询助手 | 1,500 tokens |
| 部署流程指南 | 1,200 tokens |
| **总计** | **9,200 tokens** |

### 传统方式：一次性加载

```Plaintext
初始加载：9,200 tokens（所有技能的完整内容）

每次对话：
- 写代码：9,200 tokens（但只用了代码规范技能）
- 写 PRD：9,200 tokens（但只用了 PRD 技能）
- 写文章：9,200 tokens（但只用了公众号技能）
```

**问题**：**80-90% 的 tokens 浪费了**

---

### 渐进式披露方式

#### 启动时

```Plaintext
第一级加载：5 个技能 × 100 tokens = 500 tokens
```

#### 写代码时

```Plaintext
第一级（已在内存）：500 tokens
+ 第二级（代码规范）：2,500 tokens
= 总计：3,000 tokens
```

#### 写 PRD 时

```Plaintext
第一级（已在内存）：500 tokens
+ 第二级（PRD 生成器）：1,800 tokens
= 总计：2,300 tokens
```

#### 写文章时

```Plaintext
第一级（已在内存）：500 tokens
+ 第二级（公众号写作）：2,200 tokens
+ 第三级（引用的模板文件）：800 tokens
= 总计：3,500 tokens
```

---

### 效率对比总结

| 场景 | 传统方式 | 渐进式披露 | 节省比例 |
|-|-|-|-|
| **启动时** | 9,200 tokens | 500 tokens | **94.6%** |
| **写代码** | 9,200 tokens | 3,000 tokens | **67.4%** |
| **写 PRD** | 9,200 tokens | 2,300 tokens | **75.0%** |
| **写文章** | 9,200 tokens | 3,500 tokens | **62.0%** |

**数据告诉我们**：

- 初始加载节省 **94.6%**
- 实际使用时节省 **60-75%**
- 技能越多，优势越明显

---

### 规模化效果

如果你扩展到 50 个技能：

| 方式 | 初始加载 | 可扩展性 |
|-|-|-|
| **传统方式** | \~92,000 tokens | ❌ 不可行 |
| **渐进式披露** | \~5,000 tokens | ✅ 可行 |

**结论**：渐进式披露让大规模技能配置成为可能。

## 五、实战优化技巧

理解原理后，看看如何在实际应用中优化。

### 技巧 1：合理分配三级内容

**第一级（元数据）**：

- ✅ 把最关键的识别信息放这里
- ✅ description 要精准，避免误触发
- ❌ 不要在这里放详细指令

**第二级（SKILL.md 主体）**：

- ✅ 核心工作流程
- ✅ 常用指令和规范
- ✅ 控制在 5k tokens 以内
- ❌ 不要把所有细节都塞进来

**第三级（references/）**：

- ✅ 模板文件
- ✅ 案例参考
- ✅ 详细文档
- ✅ 可以"无限"扩展

---

### 技巧 2：description 字段的艺术

**description 是触发机制的关键**。

❌ **模糊的描述**：

```YAML
description: 帮助你写代码
```

→ 太宽泛，可能在不相关场景触发

✅ **精准的描述**：

```YAML
description: 编写 TypeScript 代码时使用，确保符合团队命名规范、错误处理标准和测试要求
```

→ 明确触发条件（TypeScript）和应用场景

**最佳实践**：

1. 明确"何时使用"：什么场景下触发
2. 说明"做什么"：技能的核心功能
3. 限定范围：避免过度触发

---

### 技巧 3：何时使用 references/ 引用

**应该放 references/ 的内容**：

- ✅ 模板文件（如 PRD 模板）
- ✅ 案例参考（如示例 PRD）
- ✅ 长篇文档（如 API 文档）
- ✅ 数据文件（如数据库 schema）

**应该放在 SKILL.md 的内容**：

- ✅ 核心工作流程
- ✅ 必须执行的步骤
- ✅ 常用规则和规范

**判断标准**：

> 这个文件"每次都会用到"？放 SKILL.md  
> 这个文件"偶尔才用"？放 references/

---

### 技巧 4：常见错误与避坑

#### 错误 1：SKILL.md 过于庞大

❌ **反面案例**：

```Markdown
# PRD 生成器

## 完整流程（5000 字）
...

## 模板（3000 字）
...

## 10 个案例（15000 字）
...

总计：23,000 tokens
```

✅ **正确做法**：

- SKILL.md：核心流程（3,000 tokens）
- references/：模板 + 案例文件

---

#### 错误 2：description 不清晰

❌ **反面案例**：

```YAML
description: 这是一个很好的技能，帮助提高效率
```

→ 没说清楚什么时候用

✅ **正确做法**：

```YAML
description: 从会议记录和需求讨论中自动生成产品需求文档（PRD），适用于 SaaS 产品和功能开发
```

→ 明确了触发场景（会议记录、需求讨论）和应用领域

---

#### 错误 3：忽略第三级优化

❌ **反面案例**：

```Plaintext
skill/
└── SKILL.md (20,000 tokens 的巨型文件)
```

✅ **正确做法**：

```Plaintext
skill/
├── SKILL.md (3,000 tokens - 核心流程)
└── references/
    ├── template.md
    ├── guide.md
    └── examples/
```

## 六、总结

### 核心要点回顾

**1. 什么是渐进式披露**：

- 分三级加载技能内容
- 需要时才触发，不浪费 tokens
- 让大规模技能配置成为可能

**2. 三级加载机制**：

- 第一级：元数据（\~100 tokens，始终加载）
- 第二级：SKILL.md 主体（\~3k tokens，按需触发）
- 第三级：references/ 文件（按需加载，无限容量）

**3. Token 效率提升**：

- 初始加载节省 **90%+**
- 实际使用节省 **60-75%**
- 技能越多，优势越明显

**4. 实战优化技巧**：

- 合理分配三级内容
- 精准编写 description
- 善用 references/ 扩展
- 避免 SKILL.md 过于庞大
