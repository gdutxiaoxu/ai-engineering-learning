# 从零创建自定义技能：六步完整流程

> 作者：徐公

> 这是「Claude Code Skills 教程系列」的第 9 篇。本系列将带你从零开始，完整掌握 Claude Code Skills 这一革命性功能。

---

## 前言：为什么你需要创建自定义技能？

前面的文章里，我们聊了什么是 Skills、怎么用官方技能库，还搞懂了它的核心机制——渐进式披露。

你可能已经装了几个官方技能，觉得挺好用。但很快你会发现一个问题：**官方技能再好，也不如自己量身定制的顺手。**

### 现实场景

想象一下：

**场景一**：你是一位技术文档工程师，每次审校技术文档都要重复检查：

- 术语是否统一
- 代码示例是否可运行
- 链接是否有效
- 格式是否符合团队规范

**传统方式**：每次都要把这段要求发给 Claude，或者写在某个文档里，每次手动复制粘贴。

**有 Skills 后**：创建一个"技术文档审校"技能，之后只需要说"帮我审校这篇文档"，Claude 就会自动按照你的标准执行。

**场景二**：你是一位内容创作者，在多个平台发布内容：

- 公众号：深度长文，1500-3000字
- 小红书：种草文案，emoji + 短句
- X/Twitter：短小精悍，话题标签

**有 Skills 后**：创建三个平台的写作技能，一份核心内容，自动适配不同平台风格。

### 六步流程概览

创建自定义技能不复杂，六个步骤就够了：

1. **需求澄清**：明确技能要解决什么问题
2. **结构规划**：设计技能的目录结构
3. **编写 SKILL.md**：写技能的核心内容
4. **测试验证**：确保技能按预期工作
5. **迭代优化**：根据反馈持续改进
6. **分享与维护**：团队协作和长期维护

听着简单？确实。但每个步骤都有一些关键细节。

一步步来看。

## 第一步：需求澄清

### 1.1 明确技能要解决的问题

动手之前，先问自己三个问题：

**问题一：这个技能给谁用？**

- **仅自己使用**：可以使用个人化的术语和习惯
- **团队共享**：需要更清晰的说明和更通用的话语
- **开源贡献**：需要完善的文档和 examples

**问题二：这个技能的核心价值是什么？**

用一句话描述：**这个技能让 Claude 能够做什么？**

不好的描述：

- ❌ "帮我写代码"（太模糊）
- ❌ "提高效率"（不是技能的作用）

好的描述：

- ✅ "按照团队 TypeScript 规范编写代码"
- ✅ "将技术文档自动审校并生成修改建议"

**问题三：这个技能的触发场景是什么？**

什么时候你会希望 Claude 自动调用这个技能？

- ✅ "当我说'审校文档'时"
- ✅ "当我要写 TypeScript 代码时"
- ✅ "当我需要生成 PRD 时"

### 1.2 判断是否适合技能化

不是所有需求都适合做成技能。用这个清单判断一下：

**适合技能化的特征**：

- ✅ 重复性高：你会反复使用相同的流程
- ✅ 步骤明确：有清晰的执行步骤
- ✅ 相对稳定：不会频繁变化
- ✅ 可描述性：能用 Markdown 写清楚

**不适合技能化的场景**：

- ❌ 一次性任务：用完一次就再也不用
- ❌ 高度创造性：需要每次不同
- ❌ 依赖直觉：难以写成规则
- ❌ 频繁变化：每周都要改

### 1.3 需求澄清实例

假设我们要创建一个"技术文档审校"技能：

**问题一：给谁用？**

- 技术团队的文档工程师和开发者

**问题二：核心价值？**

- 自动检查技术文档的常见问题（术语统一、代码可运行、链接有效）

**问题三：触发场景？**

- "审校这篇技术文档"
- "检查文档质量"
- "文档review"

**是否适合技能化？**

- ✅ 重复性高：每篇文档都需要审校
- ✅ 步骤明确：有检查清单
- ✅ 相对稳定：审校标准不会频繁变化
- ✅ 可描述性：可以写成清晰的规则

**结论**：适合创建技能

## 第二步：结构规划

### 2.1 确定技能目录结构

最简单的技能，一个文件夹加一个 `SKILL.md` 就够了：

```Plaintext
tech-doc-reviewer/
└── SKILL.md
```

复杂一点的，你可能需要：

```Plaintext
tech-doc-reviewer/
├── SKILL.md              # 核心内容（必需）
├── references/           # 参考文档（可选）
│   ├── terminology.md    # 术语表
│   ├── checklists.md     # 检查清单
│   └── examples/         # 示例文档
│       ├── good-doc.md
│       └── bad-doc.md
├── scripts/              # 可执行脚本（可选）
│   └── check-links.js    # 链接检查脚本
└── assets/               # 资源文件（可选）
    └── template.md       # 文档模板
```

**何时需要额外的文件？**

| 文件夹 | 何时需要 | 示例 |
|-|-|-|
| `references/` | 内容太长，需要拆分 | 完整的代码规范、术语表 |
| `scripts/` | 需要执行代码 | 自动化检查、格式化工具 |
| `assets/` | 需要模板或资源 | PRD模板、邮件模板 |

### 2.2 规划 SKILL.md 内容框架

`SKILL.md` 是技能的核心，建议包含以下部分：

```Markdown
---
name: skill-name
description: 简洁描述技能作用和触发场景
---

# 技能名称

## Getting Started
简要说明技能用途和基本用法

## Core Workflows
核心工作流程（分步骤）

## Extended Capabilities
- **功能A**: See [feature_a.md](references/feature_a.md)
- **功能B**: See [feature_b.md](references/feature_b.md)

## Examples
具体的输入输出示例
```

### 2.3 结构规划实例

继续"技术文档审校"技能：

**目录结构**：

```Plaintext
tech-doc-reviewer/
├── SKILL.md              # 核心审校流程
└── references/           # 需要拆分的内容
    ├── terminology.md    # 常见术语表（会很长）
    ├── checklists.md     # 详细检查清单
    └── examples/         # 对比示例
        ├── good-example.md
        └── bad-example.md
```

**SKILL.md 框架**：

```Markdown
---
name: tech-doc-reviewer
description: 审校技术文档的术语一致性、代码可运行性和链接有效性
---

# 技术文档审校技能

## Getting Started
当需要审校技术文档时，使用此技能...

## Core Workflows
1. 术语一致性检查
2. 代码示例验证
3. 链接有效性检查
4. 格式规范检查

## Extended Capabilities
- **完整术语表**: See [terminology.md](references/terminology.md)
- **详细检查清单**: See [checklists.md](references/checklists.md)

## Examples
用户: 审校这篇技术文档
...
```

## 第三步：编写 SKILL.md

### 3.1 编写 Frontmatter 元数据

元数据是 Claude 判断何时加载技能的关键：

```YAML
---
name: tech-doc-reviewer
description: 当审校技术文档时使用此技能，检查术语一致性、代码可运行性和链接有效性
---
```

**编写技巧**：

**name（技能名称）**：

- 用 kebab-case（小写+连字符）
- 简洁明了，见名知意
- 避免特殊字符和空格

**description（技能描述）**：

- **说明作用**：这个技能做什么
- **说明场景**：什么时候使用
- **包含关键词**：便于 Claude 匹配

好的描述示例：

```YAML
# 好的描述
description: 当编写 TypeScript 代码时使用此技能，确保代码符合团队规范（命名、注释、错误处理）

# 分析：
# - 作用：确保代码符合规范
# - 场景：编写 TypeScript 代码时
# - 关键词：TypeScript, 代码, 规范
```

不好的描述示例：

```YAML
# 不好的描述
description: 这是一个代码规范技能

# 问题：
# - 没说明具体是什么语言
# - 没说明何时使用
# - 太泛，无法准确触发
```

### 3.2 编写主体内容

**Getting Started 部分**：

这是 Claude 首先看到的内容，要简洁明了：

```Markdown
## Getting Started

本技能用于审校技术文档的质量，主要包括：
- **术语一致性**：确保技术术语使用统一
- **代码可运行性**：验证代码示例可以执行
- **链接有效性**：检查所有链接是否可访问
- **格式规范**：符合团队文档格式要求

**触发场景**：
- "审校这篇技术文档"
- "检查文档质量"
- "文档review"
- "帮我看看这篇文档有什么问题"
```

**Core Workflows 部分**：

这是技能的核心，要给出清晰的步骤：

```Markdown
## Core Workflows

### 第一步：术语一致性检查

1. 提取文档中的所有技术术语
2. 对照术语表检查：
   - 同一概念是否使用了不同表述
   - 缩写是否在首次出现时定义
   - 专业术语是否使用正确
3. 列出所有不一致之处

### 第二步：代码示例验证

1. 找出所有代码块
2. 检查：
   - 语法是否正确
   - 是否有明显的运行时错误
   - 是否缺少必要的导入
   - 变量命名是否清晰
3. 标记可疑代码并给出修改建议

### 第三步：链接有效性检查

1. 提取所有 URL
2. 检查：
   - 链接格式是否正确
   - 是否包含测试环境链接
   - 锚点是否有效
3. 标记无效链接

### 第四步：格式规范检查

1. 检查标题层级（h1 > h2 > h3）
2. 检查列表格式
3. 检查代码块语言标识
4. 检查图片 alt 文本
```

### 3.3 添加 Examples

Examples 是确保技能正确触发的关键：

```Markdown
## Examples

### Example 1: 完整审校

**用户**: 审校这篇技术文档

**Claude**:
我将按照以下步骤审校文档：

1. **术语一致性检查**
   - 发现问题："用户"和"使用者"混用
   - 建议：统一使用"用户"

2. **代码示例验证**
   - 第3行代码块：缺少 `import React`
   - 第7行：变量名 `data1` 不够清晰

3. **链接有效性检查**
   - 所有链接格式正确
   - 发现1个测试环境链接

4. **格式规范检查**
   - 标题层级正确
   - 代码块都有语言标识

### Example 2: 针对性检查

**用户**: 帮我检查这篇文档的术语是否统一

**Claude**: （仅执行术语一致性检查）

### Example 3: 代码审查

**用户**: 这篇技术文档的代码示例能运行吗？

**Claude**: （仅执行代码示例验证）
```

**Examples 编写技巧**：

- ✅ 覆盖主要使用场景
- ✅ 展示不同的触发方式
- ✅ 给出具体的输入输出
- ✅ 至少3个示例

## 第四步：测试验证

### 4.1 基础功能测试

创建技能后，立即进行基础测试：

**测试一：技能能否被识别**

```Plaintext
测试方式：在对话中提到技能的关键词
预期结果：Claude 应该加载该技能
测试命令："帮我审校这篇技术文档"
```

**测试二：技能是否按预期执行**

```Plaintext
测试方式：提供一篇真实文档
预期结果：Claude 按照流程逐步检查
验证点：
- 是否执行了所有四个步骤？
- 检查结果是否准确？
- 建议是否有用？
```

**测试三：边界情况**

```Plaintext
测试场景：
- 空文档
- 纯代码文档
- 没有代码的文档
- 格式混乱的文档

预期：技能应该能优雅处理，不崩溃
```

### 4.2 触发准确性测试

这是最常见的调试场景：技能不被触发。

**测试方法**：

```Plaintext
尝试不同的表达：
- "审校这篇文档"
- "帮我看看文档质量"
- "检查一下这篇技术文档"
- "这篇文档有什么问题？"

记录：
- 哪些表达能触发？
- 哪些不能触发？
```

**如果不触发，检查**：

1. **description 是否包含关键词？**

   - 如果你说"审校文档"，但 description 里只有"review"，可能不匹配
   - 解决：在 description 中添加中英文关键词
2. **技能是否在正确位置？**

   - 必须在 `.claude/skills/` 目录下
   - 目录名必须是 `skill-name/SKILL.md` 格式
3. **SKILL.md 格式是否正确？**

   - frontmatter 必须是有效的 YAML
   - name 和 description 字段不能缺失

### 4.3 调试技巧

**技巧一：使用 Think Aloud**

在 SKILL.md 开头添加：

```Markdown
## 调试模式

当执行此技能时，请：
1. 说明你加载了哪个技能
2. 说明你为什么认为这个技能匹配当前任务
3. 在每个步骤开始前，说明要做什么
```

**技巧二：简化测试**

如果技能复杂，先测试简化版本：

```Plaintext
1. 先只保留 Getting Started 和一个简单步骤
2. 确认能触发后，再逐步添加功能
3. 这样可以快速定位问题
```

**技巧三：查看日志**

如果技能不工作，检查：

```Plaintext
- Claude 是否提到了技能名称？（说明已加载）
- Claude 是否执行了相关步骤？（说明已触发）
- 哪一步出了问题？（针对性修复）
```

## 第五步：迭代优化

### 5.1 根据测试结果优化 description

description 是技能的"门面"，需要持续优化：

**优化方向一：添加触发关键词**

```YAML
# 初始版本
description: 审校技术文档

# 优化后
description: 当审校技术文档、检查文档质量、文档review时使用此技能

# 添加了："检查文档质量"、"文档review"
```

**优化方向二：明确技能边界**

```YAML
# 初始版本
description: 帮助写代码

# 优化后
description: 当编写 TypeScript 代码时使用此技能，确保符合团队规范

# 明确了：
# - 只针对 TypeScript
# - 只负责规范检查，不负责功能实现
```

**优化方向三：添加排除条件**

```YAML
# 明确说明什么时候不用
description: 用于审校已完成的技术文档。不适用于：
# - 正在草稿阶段的文档
# - 非技术类文档（如营销文案）
# - 需要翻译的文档
```

### 5.2 调整内容组织

根据实际使用反馈调整：

**问题一：Claude 执行顺序不对**

```Plaintext
原因：SKILL.md 中的步骤顺序可能不够清晰

解决：
1. 使用明确的编号（1、2、3）
2. 在每个步骤前加"### Step X"
3. 强调"必须按顺序执行"
```

**问题二：Claude 遗漏某些步骤**

```Plaintext
原因：步骤描述不够明确

解决：
1. 每个步骤用"必须"、"应该"等词强调
2. 添加检查清单格式
3. 在 Examples 中展示完整流程
```

**问题三：Claude 理解偏差**

```Plaintext
原因：指令有歧义

解决：
1. 添加更多 Examples
2. 使用"不要..."明确排除错误理解
3. 提供正反对比示例
```

### 5.3 添加更多 Examples

Examples 是技能的"使用手册"，越完善越好：

**Example 类型一：完整流程**

```Markdown
### Example: 完整审校流程

**输入**: 一篇完整的技术文档

**输出**:
## 审校报告

### 1. 术语一致性 ⚠️
- 问题：第3段用"用户"，第7段用"使用者"
- 建议：统一使用"用户"

### 2. 代码可运行性 ❌
- 问题：代码块2缺少 `import`
- 修复：[提供修复后的代码]

### 3. 链接有效性 ✅
- 所有链接格式正确

### 4. 格式规范 ⚠️
- 问题：第5节标题层级错误（h3 → h2）
```

**Example 类型二：部分功能**

```Markdown
### Example: 仅检查术语

**输入**: "帮我看看术语是否统一"

**输出**:
（仅执行术语一致性检查）
```

**Example 类型三：错误处理**

```Markdown
### Example: 处理空文档

**输入**: 空文档或只有标题的文档

**输出**:
本文档内容较少，无需全面审校。
建议：补充更多技术细节和代码示例。
```

## 第六步：分享与维护

### 6.1 版本管理建议

即使是个人的技能，也建议使用版本管理：

**方法一：Git 版本控制**

```Bash
# 在项目根目录初始化 Git
git init

# 提交初始版本
git add .claude/skills/
git commit -m "添加技术文档审校技能"

# 后续修改
git commit -m "优化：增加代码示例验证步骤"
git commit -m "修复：术语检查遗漏缩写"
```

**好处**：

- 可以回滚到之前版本
- 可以查看修改历史
- 方便团队协作

**方法二：文件版本号**

在 SKILL.md 开头添加：

```Markdown
---
name: tech-doc-reviewer
version: 1.2.0
description: ...
last_updated: 2025-01-15
changelog: |
  - v1.2.0: 新增代码示例验证
  - v1.1.0: 优化术语检查逻辑
  - v1.0.0: 初始版本
---
```

### 6.2 团队共享方式

**方式一：通过项目仓库共享**

```Plaintext
项目仓库/
└── .claude/
    └── skills/
```

````Markdown
# 技术文档审校技能

## Getting Started

本技能用于系统化审校技术文档的质量，包括：
- **术语一致性**：确保技术术语使用统一，参见 [common-terms.md](references/common-terms.md)
- **代码可运行性**：验证代码示例语法正确、无运行时错误
- **链接有效性**：检查所有链接格式正确、无测试环境链接
- **格式规范**：符合团队文档格式要求，参见 [format-guide.md](references/format-guide.md)

**触发场景**：
- "审校这篇技术文档"
- "检查文档质量"
- "文档review"
- "帮我看看这篇文档有什么问题"

## Core Workflows

### Step 1: 术语一致性检查

1. 提取文档中的所有技术术语（包括缩写、专业词汇）
2. 对照 [common-terms.md](references/common-terms.md) 检查：
   - 同一概念是否使用了不同表述
   - 缩写是否在首次出现时定义（格式：`全称（缩写）`）
   - 专业术语是否使用正确
3. 列出所有不一致之处，给出修改建议

**输出格式**：
```markdown
### 1. 术语一致性检查
- ✅ 术语使用正确
- ⚠️ 第X段："用户"和"使用者"混用，建议统一使用"用户"
````

### Step 2: 代码示例验证

1. 找出所有代码块（\`\`\`标记的内容）
2. 对每个代码块检查：

   - **语法检查**：是否有明显的语法错误
   - **完整性检查**：是否缺少必要的导入或初始化
   - **命名检查**：变量名、函数名是否清晰易懂
   - **注释检查**：复杂逻辑是否有注释说明
3. 标记可疑代码并给出修改建议

**输出格式**：

```Markdown
### 2. 代码示例验证
- ✅ 代码块1（第5行）：正确
- ❌ 代码块2（第12行）：缺少 `import React`
- ⚠️ 代码块3（第20行）：变量名 `data1` 不够清晰，建议改为 `userDataList`
```

### Step 3: 链接有效性检查

1. 提取文档中的所有 URL（包括 markdown 链接和纯文本 URL）
2. 检查：

   - 链接格式是否正确（http/https、www 等）
   - 是否包含测试环境链接（localhost、127.0.0.1、测试域名）
   - 锚点链接是否有效（# 号后的内容是否存在）
3. 标记无效或可疑链接

**输出格式**：

```Markdown
### 3. 链接有效性检查
- ✅ 所有外部链接格式正确
- ⚠️ 第8段：包含测试环境链接 (http://test.example.com)，发布前需替换
```

### Step 4: 格式规范检查

根据 format-guide.md 检查：

1. **标题层级**：

   - 文档只能有1个 h1 标题
   - 标题层级不能跳跃（h1 → h3）
2. **列表格式**：

   - 同级列表使用相同标记（都是 `-` 或都是 `1.`）
   - 列表项内容对齐
3. **代码块**：

   - 所有代码块都有语言标识（如 \`\`\`javascript）
   - 代码块前后有空行
4. **图片**：

   - 所有图片都有 alt 文本
   - 图片路径正确

**输出格式**：

```Markdown
### 4. 格式规范检查
- ✅ 标题层级正确
- ⚠️ 第3节：列表标记不统一（混用 `-` 和 `*`），建议统一使用 `-`
```

### Step 5: 生成审校报告

汇总上述检查结果，生成结构化的审校报告：

```Markdown
# 技术文档审校报告

## 总体评分：B+

## 详细检查结果

### 1. 术语一致性 ⚠️
[具体问题列表]

### 2. 代码示例验证 ❌
[具体问题列表]

### 3. 链接有效性 ✅
[检查结果]

### 4. 格式规范 ⚠️
[具体问题列表]

## 修改建议优先级
1. **高优先级**（必须修改）：代码示例错误
2. **中优先级**（建议修改）：术语不统一
3. **低优先级**（可选优化）：格式微调
```

## 总结

### 核心要点回顾

创建自定义技能，六个步骤：

1. **需求澄清**：明确问题、判断价值、定义场景
2. **结构规划**：设计目录、规划内容、决定是否需要 references
3. **编写 SKILL.md**：写好 frontmatter、主体内容、Examples
4. **测试验证**：基础功能、触发准确性、边界情况
5. **迭代优化**：优化 description、调整内容、添加 Examples
6. **分享与维护**：版本管理、团队共享、持续维护

### 成功技能的特征

**好的技能**：

- ✅ description 清晰明确，包含关键词
- ✅ 步骤清晰，Examples 丰富
- ✅ 经过真实场景测试
- ✅ 持续迭代优化

**失败的技能**：

- ❌ description 模糊，无法触发
- ❌ 指令不清晰，Claude 理解偏差
- ❌ 没有 Examples，行为不一致
- ❌ 创建后从不维护
