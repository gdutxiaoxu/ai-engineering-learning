# Skills vs Sub-Agents vs MCP：你应该用哪个？

> 作者：徐公

> 这是「Claude Code Skills 教程系列」的第 3 篇。本系列将带你从零开始，完整掌握 Claude Code Skills 这一革命性功能。

---

## 一、选择困难症的烦恼

你可能有这样的经历：

听说了 Claude Code 的三种扩展方式——Skills、Sub-Agents、MCP——但每个都好像能干类似的事。

- **Skills**？可以给 Claude 加专业知识
- **Sub-Agents**？可以并行处理任务
- **MCP**？可以连接外部工具和数据

然后你就懵了。

**我到底该用哪个？**

更糟的是，三种方式好像都能用，但你不知道哪个最合适。

选错了？浪费时间配置，效果还不好，最后返工。这感觉太糟糕了。

这篇文章用最简单的方式，帮你搞懂三种机制的区别，在实际场景中选对工具。

---

## 二、三种机制快速对比

先给你一个全景图。

### 一句话定义

- **Skills**：给 Claude 装专业知识包，需要时自动调用
- **Sub-Agents**：派多个 Claude 分身并行干活
- **MCP**：给 Claude 连接外部世界的桥梁（数据库、API、工具）

### 形象比喻

想象你在经营一家餐厅：

**Skills** 就像是厨师的**专业技能档案**：

- 这位厨师擅长川菜
- 那位厨师擅长日料
- 根据客人点菜，自动调用对应厨师

**Sub-Agents** 就像是**多个厨房同时开工**：

- 前厨准备前菜
- 主厨做主菜
- 糕点师做甜品
- 三条线并行，提高效率

**MCP** 就像是**餐厅的供应链系统**：

- 连接菜市场，实时采购食材
- 连接供应商，订购厨具
- 连接外卖平台，接收订单
- 让餐厅能使用外部资源

### 核心对比表

| 特性 | Skills | Sub-Agents | MCP |
|-|-|-|-|
| **核心目的** | 专业知识、工作流程 | 并行任务、探索研究 | 外部工具和数据源 |
| **调用方式** | Claude 自动发现和加载 | 父代理显式生成和编排 | 通过 MCP 服务器工具调用 |
| **配置复杂度** | 低（只需 SKILL.md） | 中等（需要编排逻辑） | 中-高（需服务器设置） |
| **执行方式** | 串行，按需加载 | 并行，同时运行 | 按需调用外部工具 |
| **数据流向** | 内部知识传递 | 多 Agent 协作 | 内外部数据交互 |
| **最适合场景** | 领域专业知识、模板化流程 | 并行工作、多角度探索 | 外部 API、数据库查询 |
| **典型例子** | 代码规范、PRD 生成器 | 同时研究多个技术方案 | 查询数据库、调用 GitHub API |
| **学习成本** | 1-2 小时 | 2-4 小时 | 4-8 小时 |
| **维护成本** | 低 | 中 | 中-高 |

### 关键区别总结

用一句话概括：

- **Skills**：解决"知识复用"问题
- **Sub-Agents**：解决"并行效率"问题
- **MCP**：解决"外部连接"问题

理解了这个，选择就清晰多了。

## 三、Skills：专业知识打包

### 核心特点

**Skills** 是最简单、最轻量的扩展方式。

本质上？一个文件夹 + 一个 Markdown 文件。

```Plaintext
my-skill/
└── SKILL.md
```

**SKILL.md** 包含两部分：

1. **元数据**（YAML）：告诉 Claude 这个技能叫什么、什么时候用
2. **指令内容**：具体执行步骤和规范

### 典型使用场景

#### 场景1：团队代码规范

**问题**：每次写代码都要重复输入规范要求

**Skills 解决方案**：

```YAML
---
name: typescript-coding-standards
description: 编写 TypeScript 代码时自动应用团队规范
---
```

之后，当你让 Claude 写 TypeScript 代码时，它会：

- 自动识别这个技能相关
- 加载完整的规范文档
- 按照规范写代码

**效果**：一次配置，永久使用（懒人福音）

#### 场景2：文档模板自动化

**问题**：每次写 PRD、技术文档都要回忆格式

**Skills 解决方案**：

创建一个 `prd-generator` 技能，包含：

- PRD 模板
- 写作规范
- 示例参考

之后你只需要说："把会议记录整理成 PRD"

Claude 自动加载技能，按照模板生成文档。

**真实案例**：有人用这个方法，把生成 PRD 的时间从 2 小时缩短到 5 分钟。不是夸张，是真快。

#### 场景3：多平台内容创作

**问题**：一份内容要发布到多个平台，每个平台风格不同

**Skills 解决方案**：

```Plaintext
.claude/skills/
├── wechat-article-writer/    # 公众号风格
├── xiaohongshu-writer/       # 小红书风格
└── twitter-writer/           # Twitter 风格
```

每个技能包含对应平台的：

- 字数要求
- 语言风格
- 排版规范
- 标题技巧

之后你只需要说："把这个内容改编成小红书文案"

Claude 会自动加载小红书技能，生成符合风格的文案。

### Skills 的优势

✅ **零门槛**：只需要会写 Markdown  
✅ **自动触发**：Claude 根据任务自动识别和调用  
✅ **高效利用 Token**：渐进式披露机制，按需加载  
✅ **易于维护**：修改文本文件即可，无需编程  
✅ **可共享**：通过 Git 共享，团队协作方便

### Skills 的局限

❌ **无法并行**：只能串行执行  
❌ **无法连接外部**：不能访问数据库或 API  
❌ **依赖模型判断**：触发准确度依赖 description 编写（写好描述很重要）

### 什么时候用 Skills

你的任务符合以下特征时，优先考虑 Skills：

- ✅ 需要复用专业知识或流程
- ✅ 不需要访问外部数据
- ✅ 不需要并行处理
- ✅ 希望快速配置，低维护成本

## 四、Sub-Agents：并行任务专家

### 核心特点

**Sub-Agents** 是 Claude Code 的高级功能，允许你创建多个"子代理"并行工作。

每个 Sub-Agent 是一个独立的 Claude 实例，有自己的：

- 任务目标
- 上下文信息
- 执行逻辑

父代理负责：

- 创建 Sub-Agents
- 分配任务
- 汇总结果（像个项目经理）

### 典型使用场景

#### 场景1：技术方案调研

**问题**：需要从多个角度评估一个技术方案

**传统方式**（串行）：

1. 先调研性能
2. 再调研成本
3. 最后调研可维护性  
→ 总耗时：3 小时

**Sub-Agents 方式**（并行）：

```Plaintext
父代理
├── Sub-Agent 1：调研性能
├── Sub-Agent 2：调研成本
└── Sub-Agent 3：调研可维护性
```

→ 总耗时：1 小时

**效果**：3 个 Sub-Agent 同时工作，时间缩短 66%。真不是小数目。时间就是金钱。

#### 场景2：多角度代码审查

**问题**：需要从多个维度审查代码（安全性、性能、可维护性）

**Sub-Agents 解决方案**：

```Plaintext
父代理
├── Sub-Agent 1：安全性审查
├── Sub-Agent 2：性能分析
└── Sub-Agent 3：可维护性评估
```

每个 Sub-Agent 专注于一个维度，最后父代理汇总成完整报告。

#### 场景3：内容创作 A/B 测试

**问题**：需要生成多个版本的内容进行对比

**Sub-Agents 解决方案**：

```Plaintext
父代理
├── Sub-Agent 1：生成保守版本
├── Sub-Agent 2：生成激进版本
└── Sub-Agent 3：生成平衡版本
```

三个版本同时生成，你可以选择最满意的。

### Sub-Agents 的优势

✅ **并行加速**：多个任务同时进行，大幅缩短时间  
✅ **专业分工**：每个 Agent 专注一个方向  
✅ **多角度探索**：同时尝试不同方案，对比选择  
✅ **灵活编排**：可以动态创建和调整

### Sub-Agents 的局限

❌ **复杂度高**：需要设计编排逻辑  
❌ **成本较高**：多个 Agent 同时运行，Token 消耗大（钱包会疼）  
❌ **结果整合**：需要汇总多个 Agent 的输出  
❌ **调试困难**：并行执行的问题较难排查

### 什么时候用 Sub-Agents

你的任务符合以下特征时，考虑 Sub-Agents：

- ✅ 可以拆分成多个独立的子任务
- ✅ 子任务之间没有强依赖关系
- ✅ 需要从多个角度探索
- ✅ 时间敏感，希望并行加速

## 五、MCP：外部世界的桥梁

### 核心特点

**MCP（Model Context Protocol）** 是 Claude 连接外部世界的标准协议。

通过 MCP，Claude 可以：

- 查询数据库
- 调用 API
- 读写文件系统
- 使用外部工具

**架构**：

```Plaintext
Claude Code
    ↓
MCP Client
    ↓
MCP Server
    ↓
外部资源（数据库、API、文件系统...）
```

### 典型使用场景

#### 场景1：数据库查询

**问题**：需要查询公司数据库获取实时数据

**MCP 解决方案**：

1. 部署 MCP Server，连接数据库
2. Claude 通过 MCP 调用查询工具
3. 返回结果供 Claude 使用

**示例**：

你："查询本月新用户的订单数量"

Claude：

1. 识别需要查询数据库
2. 通过 MCP 调用查询工具
3. 执行 SQL：`SELECT COUNT(*) FROM orders WHERE user_created_at >= '2026-01-01'`
4. 返回结果："本月有 1,247 个新用户下单"

#### 场景2：GitHub API 集成

**问题**：需要分析 GitHub 仓库的 issue 趋势

**MCP 解决方案**：

通过 GitHub MCP Server，Claude 可以：

- 获取仓库列表
- 读取 issue 详情
- 分析贡献者活动
- 生成统计报告

#### 场景3：文件系统操作

**问题**：需要批量处理本地文件

**MCP 解决方案**：

通过 filesystem MCP Server，Claude 可以：

- 遍历目录结构
- 读取文件内容
- 批量重命名
- 生成文件报告

### MCP 的优势

✅ **连接外部**：突破 Claude 的沙箱限制  
✅ **实时数据**：可以获取最新的外部信息  
✅ **功能扩展**：集成任意外部工具  
✅ **标准化**：统一的协议，生态丰富

### MCP 的局限

❌ **复杂度高**：需要部署和配置 MCP Server  
❌ **安全风险**：连接外部资源需谨慎（别连数据库炸了）  
❌ **依赖性**：需要额外的服务和维护  
❌ **学习曲线**：需要理解 MCP 协议和 Server 开发

### 什么时候用 MCP

你的任务符合以下特征时，考虑 MCP：

- ✅ 需要访问外部数据源（数据库、API）
- ✅ 需要操作外部系统（文件系统、云服务）
- ✅ 需要实时数据
- ✅ 有技术能力部署和维护 Server

## 六、如何选择：决策树

### 三步选择法

#### 第一步：问自己三个问题

1. **是否需要访问外部数据（数据库、API）？**

   - 是 → 考虑 MCP
   - 否 → 进入下一步
2. **是否可以并行执行多个独立任务？**

   - 是 → 考虑 Sub-Agents
   - 否 → 进入下一步
3. **是否需要复用专业知识或流程？**

   - 是 → 使用 Skills
   - 否 → 可能不需要任何扩展机制

#### 第二步：对照典型场景

| 你的需求 | 推荐方案 |
|-|-|
| 代码规范、文档模板、写作风格 | Skills |
| PRD 生成、测试用例生成、代码审查 | Skills |
| 多方案并行调研、多角度审查 | Sub-Agents |
| A/B 测试、批量内容生成 | Sub-Agents |
| 查询数据库、调用 API、操作文件系统 | MCP |
| 集成第三方服务（GitHub、Jira） | MCP |

#### 第三步：考虑成本和复杂度

| 维度 | Skills | Sub-Agents | MCP |
|-|-|-|-|
| **学习成本** | 低（1-2 小时） | 中（2-4 小时） | 高（4-8 小时） |
| **配置时间** | 5-10 分钟 | 30-60 分钟 | 2-4 小时 |
| **维护成本** | 低 | 中 | 中-高 |
| **Token 消耗** | 低 | 高 | 中 |

**建议**：

- 先从 Skills 开始（最简单）
- 熟悉后再尝试 Sub-Agents
- 有明确需求时再部署 MCP（别一开始就上大杀器）

### 决策树图

```Plaintext
开始
    ↓
需要访问外部数据（数据库/API）？
    ↓ 是
使用 MCP ←───────────────────────────┐
    ↓ 否                              |
可以并行执行多个独立任务？             |
    ↓ 是                              |
使用 Sub-Agents ←─────────────────────┤
    ↓ 否                              |
需要复用专业知识或流程？              |
    ↓ 是                              |
使用 Skills ←─────────────────────────┤
    ↓ 否                              |
不需要扩展机制                        |
    ↓                                 |
结束 ←────────────────────────────────┘
```

### 组合使用案例

在实际项目中，三种机制常常组合使用。

**案例：自动化 PRD 生成系统**

```Plaintext
系统架构：
├── Skills（知识层）
│   ├── prd-template-skill        # PRD 模板
│   ├── writing-style-skill       # 写作规范
│   └── domain-knowledge-skill    # 领域知识
│
├── Sub-Agents（并行层）
│   ├── Agent 1：分析竞品
│   ├── Agent 2：调研用户需求
│   └── Agent 3：技术可行性评估
│
└── MCP（数据层）
    ├── Database MCP：查询历史数据
    ├── GitHub MCP：关联技术 issue
    └── Notion MCP：同步到知识库
```

**工作流程**：

1. Sub-Agents 并行调研（竞品、用户、技术）
2. MCP 查询历史数据和关联信息
3. Skills 应用 PRD 模板和写作规范
4. 生成完整的 PRD 文档

**效果**：从需求到 PRD，30 分钟完成（传统方式需要 2-4 小时）。这效率提升，真不是吹的。

## 七、实战选择指南

### 场景1：开发者建立代码规范

**需求**：团队需要统一的代码规范

**推荐**：Skills

**理由**：

- 代码规范是静态知识，不需要外部数据
- 不需要并行执行
- Skills 可以自动触发，使用方便

**实施**：

```Plaintext
.claude/skills/
└── coding-standards/
    ├── SKILL.md
    ├── references/
    │   ├── typescript-guide.md
    │   ├── react-best-practices.md
    │   └── naming-conventions.md
    └── examples/
        └── good-code-sample.ts
```

---

### 场景2：内容创作者多平台发布

**需求**：一篇文章发布到公众号、小红书、Twitter

**推荐**：Skills

**理由**：

- 每个平台的风格规范是静态知识
- 可以创建多个技能，自动适配
- 维护成本低

**实施**：

```Plaintext
.claude/skills/
├── wechat-writer/
│   └── SKILL.md  # 公众号风格（1500-3000字，深度内容）
├── xiaohongshu-writer/
│   └── SKILL.md  # 小红书风格（500-1000字，种草文案）
└── twitter-writer/
    └── SKILL.md  # Twitter 风格（<280字，短平快）
```

---

### 场景3：技术团队多方案选型

**需求**：从多个技术方案中选择最优解

**推荐**：Sub-Agents

**理由**：

- 需要并行调研多个方案
- 每个方案评估相对独立
- 并行可以大幅节省时间

**实施**：

```Plaintext
父代理创建 3 个 Sub-Agents：
├── Agent A：评估方案 A（性能、成本）
├── Agent B：评估方案 B（可维护性、生态）
└── Agent C：评估方案 C（学习曲线、风险）

最后汇总：对比分析 + 推荐
```

---

### 场景4：数据分析师查询业务数据

**需求**：定期查询数据库，生成分析报告

**推荐**：MCP

**理由**：

- 必须访问外部数据库
- 需要实时数据
- Skills 和 Sub-Agents 都无法解决（这活儿只能 MCP 干）

**实施**：

```Plaintext
部署 PostgreSQL MCP Server
├── 连接到业务数据库
├── 配置查询权限
└── 创建常用查询模板

Claude 可以：
- "查询本月销售额 TOP10 的产品"
- "对比去年同期增长情况"
- "生成可视化报告"
```

---

### 场景5：创业公司快速原型开发

**需求**：快速验证产品想法，需要集成多个服务

**推荐**：Skills + MCP 组合

**理由**：

- Skills：提供产品开发流程和最佳实践
- MCP：连接外部服务（支付、地图、推送等）
- 组合拳才是王道

**实施**：

```Plaintext
Skills：
├── startup-mvp-skill        # MVP 开发流程
└── user-research-skill      # 用户研究方法

MCP：
├── Stripe MCP：支付集成
├── Google Maps MCP：地图服务
└── Firebase MCP：用户认证
```

## 八、总结与下篇预告

### 核心要点回顾

**三种机制的本质区别**：

- Skills：知识复用
- Sub-Agents：并行效率
- MCP：外部连接

**选择决策**：

- 需要外部数据 → MCP
- 可以并行执行 → Sub-Agents
- 知识复用 → Skills

**成本考虑**：

- Skills：最简单，优先尝试
- Sub-Agents：中等复杂度，合适场景很有价值
- MCP：最复杂，但能解决独特问题

**组合使用**：

- 实际项目中，三种机制常常配合
- 各司其职，发挥最大价值
